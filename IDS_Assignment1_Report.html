02

What we're building

A modern, web-based IT Help Desk and Ticketing Management System designed to streamline how technical support works inside a company. The idea is straightforward: employees submit support requests through a clean interface, and IT agents, managers, and admins manage everything from a centralized dashboard — assigning tickets, updating statuses, tracking resolution times.

The project is intentionally scoped to simulate a real enterprise environment. By the end of the internship it should cover frontend development, RESTful APIs, relational database design, authentication with role-based access control, reporting, and a first pass at AI-assisted features like category suggestions and smart replies.

03

Who uses the system

Four distinct roles, each with its own view of the world:

Admin

Full system access — user management, roles, categories, settings, and reports.

IT Support Agent

Manages, assigns, updates, comments on, and resolves tickets.

Employee

Creates support tickets, tracks progress, and replies to agent comments.

Manager

Monitors team performance and reports without full admin access.

06

System modules

Authentication & User Management

Login, registration (or admin-created accounts), password reset, profile management, and full role-based access control. Every API route will be protected — Admins, Agents, Employees, and Managers each see a different version of the system. Sessions are token-based using JWT.

Ticket Management

The core of the system. Employees create tickets, agents work them, everyone can track history. Tickets carry a category (Hardware, Software, Network, Email, Access Request, Other), a priority, and a status.

Priority

Low Medium High Critical

Status

Open In Progress Pending Resolved Closed

Assignment & Workflow

Agents can be assigned tickets manually, and reassigned when needed. Every action leaves an audit trail — assignment history, status changes, internal notes — so nothing is lost. Automatic assignment is planned for later once categories and teams stabilize.

Notifications

In-app alerts for ticket updates, comments, and @mentions. Email notifications and real-time updates via SignalR are planned for Week 5 as the system matures.

Dashboard & Reporting

Summary widgets (open, resolved, pending counts), tickets by category and priority, and agent performance charts. Monthly reports, average resolution time, and optional SLA tracking come in Week 6. Export to PDF or Excel is part of that same milestone.

File Attachments

Employees can attach screenshots, logs, or documents to their tickets. The backend will validate file size and type before storing. Downloads are served securely.

Knowledge Base (optional/advanced)

Searchable FAQ articles organized by category. Articles require admin approval before going live. We've included this in the database schema now so the system can grow into it without requiring a migration later.

AI Features (planned for Week 6)

Automatic ticket categorization, priority suggestions, and suggested replies using the OpenAI API. An AI chat assistant for employees — to try resolving issues before creating a ticket — is also on the roadmap. These are baked into the architecture now even though they won't be built until later.

07

Database design

The full PostgreSQL schema is in database/schema.sql. We covered everything the brief asked for and went further where it made practical sense.

Core tables

  • Users, Roles, Departments
  • Tickets, Categories, Priorities, Statuses
  • TicketComments, TicketAttachments
  • Notifications, ActivityLogs

Additional tables we added

  • SupportTeams and TeamMembers — needed for assignment workflow
  • TicketAssignmentHistory — full audit trail of reassignments
  • KnowledgeBaseArticles — future-proofing the schema
  • AISuggestions — stores AI-generated hints linked to tickets

The SQL file also includes seed data for roles, priorities, statuses, departments, support teams, and categories so the system is usable from the first run.

The ERD is at diagrams/erd.mmd. Key relationships: one role to many users, one department to many users, one employee to many tickets, one ticket to one category/priority/status, and one ticket to many comments, attachments, notifications, AI suggestions, and activity logs.

08

Application architecture

The full architecture diagram lives in diagrams/application_architecture.mmd. The flow is clean and standard:

React frontend → ASP.NET Core Web API controllers → Application services → Entity Framework Core → PostgreSQL

On the backend, controllers handle HTTP, DTOs define request/response shapes, services hold business logic, and EF Core manages database access. Authentication middleware protects every non-public route.

On the frontend, each screen is a named Page component. Shared UI is broken into reusable Components. Service modules handle API calls. Route guards enforce role-based access — an employee can't reach the agent queue, and a regular user can't touch the admin console.

Initial API endpoint plan

These endpoints represent the first implementation target — not an exhaustive final list.

Method Endpoint What it does
POST/api/auth/loginAuthenticate a user, return JWT.
POST/api/auth/registerRegister or create a new user account.
GET/api/ticketsList tickets — results filtered by caller's role.
POST/api/ticketsCreate a new ticket.
GET/api/tickets/{id}Fetch full ticket detail.
PATCH/api/tickets/{id}Update status, priority, category, or assignment.
POST/api/tickets/{id}/commentsAdd a comment or internal note.
POST/api/tickets/{id}/attachmentsUpload a file attachment.
GET/api/notificationsList notifications for the current user.
PATCH/api/notifications/{id}/readMark a notification as read.
GET/api/reports/summaryDashboard summary data.
GET/api/categoriesList all ticket categories.
GET/api/admin/usersAdmin — list all users.
POST/api/admin/usersAdmin — create a new user.
GET/api/knowledge-baseSearch knowledge base articles.
POST/api/ai/ticket-suggestionsAI category, priority, and reply hints (Week 6+).
